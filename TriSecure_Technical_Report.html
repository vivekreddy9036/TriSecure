What We Built

TriSecure is a fully self-contained, tamper-resistant electronic voting terminal with three-factor security — no cloud, no network dependency, designed for electoral-grade security with audit trails, blockchain-style vote chaining, and biometric deduplication.

📡 Factor 1 — NFC Card

Physical token verification via PN532 NFC reader (SPI). Voter UUID written to NTAG2xx cards using XOR encryption derived from a SHA-256 key.

👤 Factor 2 — Face Biometrics

Live face capture with Haar Cascade detection + MobileFaceNet 512-D embedding extraction. 3-sample averaging for robust enrollment.

🔑 Factor 3 — Session Token

One-time cryptographically secure session token (60-second TTL). Consumed on vote cast; forces re-authentication on every voting action.

Security Architecture

Five-Stage Authentication Pipeline

1
NFC UID Format Validation Verify card UID format before any database lookup
2
Voter Database Lookup Indexed O(1) NFC UID → Voter record resolution
3
Eligibility Check Reject if has_voted = true — enforces one vote per voter
4
Live Face Biometric Match Captured embedding vs. stored template — cosine similarity ≥ 0.55 required
5
Session Token Issuance 60-second one-time token (secrets.token_urlsafe(32)) — consumed on vote cast

Database Schema (SQLite)

-- Voters Table
CREATE TABLE voters (
    id           TEXT PRIMARY KEY,
    name         TEXT NOT NULL,
    nfc_uid      TEXT UNIQUE NOT NULL,        -- indexed, O(1) lookups
    face_embedding BLOB,                      -- encrypted embedding
    has_voted    INTEGER DEFAULT 0,
    created_at   TEXT,
    updated_at   TEXT
);

-- Votes Table  (append-only ledger)
CREATE TABLE votes (
    sequence       INTEGER PRIMARY KEY AUTOINCREMENT,
    vote_id        TEXT NOT NULL UNIQUE,
    voter_id       TEXT NOT NULL,
    candidate      TEXT NOT NULL,
    timestamp      TEXT NOT NULL,
    previous_hash  TEXT NOT NULL,
    current_hash   TEXT NOT NULL               -- SHA256(candidate+ts+prev_hash)
);

-- Audit Events Table  (immutable compliance log)
CREATE TABLE audit_events (
    sequence    INTEGER PRIMARY KEY AUTOINCREMENT,
    event_id    TEXT NOT NULL UNIQUE,
    event_type  TEXT NOT NULL,
    voter_id    TEXT,
    timestamp   TEXT NOT NULL,
    status      TEXT NOT NULL,                -- SUCCESS | FAILURE | WARNING
    message     TEXT NOT NULL,
    details     TEXT                            -- JSON structured metadata
);

Camera Optimizations for Raspberry Pi

MJPEG codec avoids slow YUYV→BGR conversion · Low 320×240 resolution reduces USB bandwidth · Multiple grab() calls flush stale kernel-buffered frames · 15 fps sufficient for reliable face detection.
